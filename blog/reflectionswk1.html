Name 5 commands you used, and what they do.

  • 'cd' command allows you to change directory, 'cd ..' allows you to navigate up a level or directory in the heiarchy of the computers filing management system.For example, if you are in the users profile directory you will have the tilde (~) present and if you type into the commandline cd desktop the commandline will take you to the desktop directory.
  • 'mkdir' command allows you to make a directory, or make another folder.For example if you want to make a directory for the projects folder you can write 'mkdir projects', and whatever directory you are In will make a directory inside that called reflections.
  • 'mv' command allows you to rename your directory, or move as it is called to another location in the file managemnent system of your PC.
  • 'ls' command is used to list all the files (including information, like file type) and directories(folders) within the current file system; which the commandline is accessing.
  • 'pwd' command stands for "print working directory". You type in that command into the commandline and it will give you the exact file path for the file or (directory) folder you are currently accessing.
  • 'touch' command is used to create a new file, it's best to have the commandline situated in the directory you wish for the file to be located.

Version Control with Git

What's the difference between git and GitHub?

Git is localised to your PC and GitHub is cloud based where it can be accessed my multiple users or devices. I can only access Git on my Mac desktop, whereas I can access my GitHub content from my smartphone and laptop also on different wireless internet connections, keep in mind the internet is a giant network. But your local PC is isolated from the network in the sense that Git only saves content on your physical device. It's not until you share data from Git to GitHub via pushing commits in the terminal or by manually uploading them, that they are then widely acessible amongst different devices or by others user ie. your team mates on a group project.

Can you think of an analogy to describe them?

My grand mother had the most amazing chocolate chip cookie recipe, she had it written on a bit of paper, kept in a small box on the shelf in the kitchen. It wasn't until my mother took a photo, typed it out in word, printed and laminated copies to share amongst our whole family that it was widely accessible. From there the digital copy got tweaked over the years, improved, my mother said. Once we had a digital copy of Grans receipe, it was easy to share, tweak and keep it safe. Git is the origional local version or the stained bit of paper, with the recipe hand written on it living in the box on the shelf. GitHub is the digital version my mum created for the whole family to enjoy.

What is a Git workflow?

GitHub workflow or flow as it is sometimes called, is a branch based workflow. GitHub Actions is a continuous integration and continuous delivery platform that allows you to automate your build, test, and deployment pipeline. You can create workflows that build and test every pull request to your repository, or deploy merged pull requests to production. A GitHub workflow is a configurable automated process made up of one or more jobs.

Another way to describe a GitHub workflow is that can integrate the important updates of a collection of all the versions of the project along with some special information - Show what order the changes occurred; A description of each change; who was responsible for each change and this can all be viewed.

Branch, Pull, Merge

What is main?

Main or orgin, is the default name new repositories get when created in GitHub and then when new branches are created by users, they can customise the name.

Why create a Branch?

GitHub exists to allow for version control and collaboration of multiple team members or users on a single project. The branches are different versions of the origion or main version. Creating the branches to work separately on the same project allows for the project lead to ensure security of the project, to prevent possible headaches. As sometimes new updates or features will actually need to be rolled back. Which is alot easier when you just need to reload a previous commit or saved point in the project.

What are some examples of when you would fork?

A fork is a copy of a repository on GitHub owned by a different user. Forking a repository allows you to freely experiment with changes without affecting the original project. Some examples of where you might fork include when you would like to propose changes to someone else's project or to use someone else's project as a starting point for your own idea.

Some examples include when you want to create your own version of the repo by forking it to your account. Such as the reflections repo, so you can create your own version. Or, the Dev Academy Ko Wai Koe repo, so you can introduce yourself in Maori to the main. It could be any project you want to work on without impacting the main project.

Thinking like a programmer

What is your process so far for solving problems?

After going through the resources in Sprint 1, I feel more confident attacking the challenges ahead of me. Ultimiately it boils down to:

Building habits - creating an environment for myself that gets me motivated to do the things I set out to do that day. Building activities in a chain sequence (micro habits) throughout the day to ensure I take care of myself, stay on task and bases are covered to avoid distractions or interruptions.

Undertsand - Understand exactly what is being asked before starting. And if I'm unsure, approach from a new angle, try seeing it with a fresh set of eyes and after X amount of time try asking for help.

Plan - create a framework for myself to follow. Like writting an essay there is an intro, body and conclusion. It is the same with any project. And nicely Foudations has a plan, with timeboxes built into it.

Divide - Split the work into smaller problems to be solved one at a time, preferrable in chronological order but one you can get back to if you are stuck on it for too long. Sometimes it's best to do the easiest step first or eat the frog and do the hardest. All dependant of the project or task at hand though.

Get Unstuck - get a second opinion, get a perspective, have a break, look at it again, try again. All of those thing until it works. If all else fails? Start again.

Refractor - See if there is any way to improve your solution, make it more efficent, have less redundant lines, use less words to get to the point.

Reflect - Think about what you have done so far, what you had a problem with and what you have learned from your experience and hope that you can improve the next time you do somthing similar.

Practice - Everyone learns better by doing, all the time, repeat the same motion. It's like exercising. if you never go running you'll never be able to run 10km, start small, do it often and build up your fitness and tolerance.
